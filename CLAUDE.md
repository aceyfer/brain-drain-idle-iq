# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Brain Drain: Idle IQ — a satirical idle-clicker Unity game (AdVenture Capitalist-style pure positive accumulation). The player is framed as the evil mogul running the operation: tap to harvest "Brain Power" (soft currency) from the population; cumulative Brain Power earned drives an "Idiocracy Rank" (cosmetic title + diorama backdrop). **Lore layer (added 2026-06-21, see "Illumisnotti narrative/economy rewrite" below for the full system):** the Illumisnotti are the game's shadow-government villains — cowardly legacy-wealth elites hiding in bunkers, calling themselves smart while just being old-money dumb. The player rises against them, then ironically *becomes* them via "The Snotting" (Rebirth, reflavored in player-facing text only). COGS roasts them constantly in narrator lines. Brain Power is spent on buildings that produce passive income (BPPS) and bump the player's `PlayerIQ`, and can also be spent on infrastructure to raise `PlayerIQ` directly. `PlayerIQ` starts at 100 and scales infinitely upward while the app is running — no live decay loop (an earlier `IQDecaySystem` design did continuously decay IQ toward 0; that was fully replaced). **It does decay once, offline, on load**: a deliberate, explicitly-confirmed "return to play" hook added later — see `PlayerIQManager`/`SaveManager` below — so "no decay" only describes runtime behavior, not the full picture. The economy is three connected tiers: **Brain Power** (primary, tap/idle income) → **Cash** (secondary, produced by the Underground Economy building, CPS paid out on the same tick) → **Points** (tertiary, Cash converted on demand or auto-converted every 10s) — Brain Power funds buildings and PlayerIQ; Cash/Points are a separate, additive economy layered on top, not a replacement. A Rebirth loop resets current Brain Power and building tiers in exchange for permanent, stacking bonuses to all three currency tiers' multipliers/rates **and** to the tap payout multiplier (added 2026-06-21, see the rebalance section below). A `RandomEventManager` injects periodic satirical chaos events, a `DialogueManager` narrates milestones in degrading tone, and a `SaveManager` persists everything to disk across three separate save triggers.

- Engine: Unity `6000.4.8f1`, Universal Render Pipeline (2D), target platform iOS (portrait).
- No `.asmdef` files — all scripts compile into the default `Assembly-CSharp` assembly.
- No automated tests currently exist, despite `com.unity.test-framework` being a package dependency.

## Working with this repo

This is a Unity project opened/built through the Unity Editor, not a CLI-driven codebase — there is no command-line build/lint/test workflow set up. Changes to scene/prefab/asset wiring (`.unity`, `.prefab`, `.asset` YAML files) generally need to be made or verified inside the Unity Editor; hand-editing the YAML is error-prone for anything beyond tuning serialized numeric fields (see "Editing ScriptableObject data" below). C# script changes under `Assets/_Game/Scripts` can be edited directly.

## Architecture

All gameplay code lives under `Assets/_Game/Scripts`, split into `BrainDrain.Core` (simulation/state) and `BrainDrain.UI` (presentation). Everything is wired together through Unity Inspector references and runtime `FindAnyObjectByType` fallbacks rather than dependency injection — when adding a new system, follow the existing pattern of a serialized field that falls back to `GameManager.Instance` lookups.

**`GameManager`** (`Core/GameManager.cs`) is the central hub: a thread-safe singleton (`GameManager.Instance`) created with `[DefaultExecutionOrder(-100)]` so it initializes before dependents. It owns:
- The single global simulation tick: `InvokeRepeating` fires `OnSecondTick` once per second. Every system that needs per-second logic (currency idle income, the random event timer) subscribes to this instead of running its own timer.
- `OnGameInitialized`, fired once from `Start()` after the tick loop is running — other systems use this (instead of their own `Start()`) to do initial UI/state sync, since it guarantees `GameManager` and its core references are ready.
- `RankDefinition[]` — ordered (ascending threshold) array of `{rankName, threshold}` mapped against cumulative Brain Power via `GetRankName(double)`. The same threshold list semantically drives both the HUD rank text and `DioramaManager`'s active-diorama index, but each consumer recomputes the index independently by walking the same array — there's no shared "current rank index" state.

**`CurrencyManager`** (`Core/CurrencyManager.cs`) tracks three connected currency tiers plus premium Neurons:
- **Brain Power**: `BrainPower` (spendable, `double`), `CumulativeBrainPower` (lifetime earned, only ever increases, drives ranks/dioramas/building unlocks), and `_rebirthMultiplier` (permanent income multiplier, starts at 1.0). `AddIdleBPPS` accumulates passive income from buildings; payout happens once per second on `GameManager.OnSecondTick` — the *only* production pathway (`UpgradeManager` used to independently recompute BPPS every frame in its own `Update()`; removed in favor of registering each purchase's BPPS once via `AddIdleBPPS`). `SpendBrainPowerOnInfrastructure(cost)` spends Brain Power and converts the cost 1:1 into `PlayerIQManager.PlayerIQ`. **Idle BPPS payout is scaled by `GetIQProductionMultiplier()`** (added 2026-06-21 — `PlayerIQ / 100`, clamped to a max of 1) before being credited each tick — see the rebalance section below; tap income is unaffected by this, only idle production.
- **Cash**: `CurrentCash`, `CashPerSecond` (produced only by the Underground Economy building currently; paid out on the same `OnSecondTick`, scaled by `CashMultiplier` **and** the same `GetIQProductionMultiplier()`), `CashMultiplier` (starts 1.0, permanent, bumped by Rebirth). `AddCashPerSecond`/`AddCash` mirror the BPPS pattern exactly.
- **Points**: `CurrentPoints`, `PointsConversionRate` (Cash→Points rate, starts 0.1, permanent, bumped by Rebirth). `ConvertCashToPoints(amount)` clamps to whatever Cash is actually available (never refuses outright, just converts less) and fires `OnCashChanged`/`OnPointsChanged` (both balances change) plus `OnCashConverted(double convertedAmount)` — a plain C# event, only on an actual conversion, not a no-op call — added purely for `DialogueManager`'s `CashConverted` narrator trigger. Note `OnCashConverted` also fires from the 10s auto-convert tick (same method, both paths), not exclusively the CONVERT button; harmless today since no UI exposes that toggle yet. `AutoConvertCash` (bool) — if true, all current Cash auto-converts every 10s, folded into the existing `OnSecondTick` handler rather than a second timer; no UI currently exists to toggle this on, only Inspector/code. `SpendPoints(amount)`/`CanAffordPoints(amount)` mirror `SpendBrainPower`/`CanAffordBrainPower` exactly (validate-then-spend, returns false on insufficient funds) — added for `WorldRestorationManager` (see below), the first thing that actually spends Points.
- `OnBrainPowerChanged`/`OnCumulativeBrainPowerChanged`/`OnNeuronsChanged`/`OnFirstBrainPowerEarned` are plain C# `event Action<T>`, matching every other event in the codebase. **`OnCashChanged`/`OnPointsChanged` are `UnityEvent<double>` subclasses instead** (`CashChangedUnityEvent`/`PointsChangedUnityEvent`) — an explicit, deliberate deviation from the class's own convention per spec, subscribed via `AddListener`/`RemoveListener` rather than `+=`/`-=`.
- `ExecuteRebirth(multiplierBonus, cashMultiplierBonus, pointsConversionRateBonus)` zeroes `BrainPower`/`CumulativeBrainPower`/idle BPPS and permanently bumps all three rebirth bonuses; Cash/Points balances themselves are untouched by Rebirth. `LoadState(...)` (8 params) restores everything directly from a save, bypassing the multiplier scaling `AddBrainPower`/`AddCash` would otherwise apply. `OnFirstBrainPowerEarned` fires exactly once, the first time Brain Power is ever earned (in practice, the player's first tap) — `LoadState` marks it already-consumed when restoring a save with existing progress, so it can't falsely re-fire after a reload.

**`PlayerIQManager`** (`Core/PlayerIQManager.cs`, formerly `IQDecaySystem` then `WorldProgressionManager`) tracks `PlayerIQ` (`float`, starts at **100**) — positive accumulation while the app is *running*, driven by `ModifyPlayerIQ(delta)`, clamped at a minimum of zero. Fed by `CurrencyManager.SpendBrainPowerOnInfrastructure`, `UpgradeManager.TryBuyBuilding` (flat `+1` per building level purchased, any tier), and `RandomEventManager.ApplyEventEffects` (`BrainRotEventData.playerIQImpact`, which can be negative). `OnIQMilestoneCrossed` fires every time `PlayerIQ` crosses a 1000-point boundary (used by both `HUDController`'s celebration flash and `DialogueManager`'s `IQMilestone` trigger — two independent trackers using the same interval by convention, not a shared implementation).

**Offline IQ decay** *(explicitly confirmed, reintroduced 2026-06-21 — distinct from the old always-on `IQDecaySystem` design, which stays fully removed)*: `LoadStateWithOfflineDecay(restoredPlayerIQ, lastActiveUtc)` — called once from `SaveManager.ApplyLoadedDataToSystems()` instead of plain `LoadState` — linearly decays IQ toward a floor of **60** based on real-world hours elapsed since the last save, reaching the floor at **8 hours** offline and no further past that. Never runs while the app is open; `LoadState(float)` itself stays a pure no-transformation restore, used elsewhere with no transformation needed. If IQ actually dropped, it fires `OnOfflineDecayApplied(float amountLost)` — deliberately deferred one frame via a coroutine, since `SaveManager` runs at `DefaultExecutionOrder -200` and would otherwise fire before `DialogueManager`'s/`DialogueDisplayUI`'s default-order `Start()` calls have subscribed, losing the message. `DialogueManager` subscribes to this and fires its `OfflineDecayReturn` narrator line through the normal pool (kept as a plain `PlayerIQManager` event rather than calling `DialogueManager` directly, the same Core/Systems decoupling as `OnIQMilestoneCrossed`). `RestoreIQFromTap()` (called from `PlayerTapHandler.OnTap()`) adds a small flat amount per tap, but **only** while `PlayerIQ < 100` and clamped at 100 — once back at baseline, taps stop touching IQ again, so this is purely a bounded post-decay recovery mechanic, not a permanent change to tap's relationship to IQ. The last-save timestamp itself is persisted as `SaveManager.PlayerData.lastActiveUnixSeconds` (a `long` Unix-seconds value, not a `DateTime` — `JsonUtility` doesn't serialize `DateTime`'s internal fields).

**`UpgradeManager`** (`Core/UpgradeManager.cs`) owns building ownership (`Dictionary<string,int>` levels keyed by `buildingName`, not by `BuildingData` reference) and purchase validation (`unlockCumulativeBrainPower` gate against `CurrencyManager.CumulativeBrainPower`, exponential cost via `baseCost * costMultiplier^level` — cost is always paid in Brain Power, even for the Cash-producing building). Each successful `TryBuyBuilding` call registers that level's `baseBrainPowerPerSecond` via `AddIdleBPPS(...)` *and* `baseCashPerSecond` via `AddCashPerSecond(...)` (a no-op for the 6 BPS-only buildings, whose `baseCashPerSecond` defaults to 0), and bumps `PlayerIQManager.PlayerIQ` by a flat amount — there is no per-frame recompute loop. `OnBuildingsChanged` fires after a successful purchase for UI refresh; `OnBuildingPurchased(BuildingData)` fires with the specific building (used by `DialogueManager`). `ResetBuildings()` clears ownership (used by Rebirth) but relies on `CurrencyManager.ExecuteRebirth` to also zero idle BPPS (Cash/CPS are *not* zeroed by Rebirth, matching that Cash/Points are untouched by it). `LoadBuildingLevels(...)` restores ownership from a save and re-derives both idle BPPS and CPS by replaying `AddIdleBPPS`/`AddCashPerSecond` per restored level. Current 7 buildings and their real `unlockCumulativeBrainPower` gates: `TheLiteralLibrary`=0, `DoomscrollEngine`=0, `PodcasterSoundboard`=10,000, `CryptoBroCompound`=65,000, `RealityTVSyndicate`=185,000, `BrainRotThinkTank`=725,000, `UndergroundEconomy`=500 (these are tuned design values, not each building's `baseCost` — a recurring point of confusion in task requests). `UndergroundEconomy` is the only building with `baseBrainPowerPerSecond: 0` / `baseCashPerSecond: 0.5` — it's the bridge into the Cash tier, not another BPS producer.

**`BuildingData`** (`Core/BuildingData.cs`) is a `ScriptableObject` (`CreateAssetMenu` under `BrainDrain/Building Data`) holding the authoring data for one building type. Instances live as `.asset` files in `Assets/_Game/Buildings/`. `UpgradeManager.buildingTemplates` is a manually-populated list of these assets in the Inspector — adding a new building means creating a new asset here *and* adding it to that list.

**`DioramaManager`** (`Core/DioramaManager.cs`) swaps which of N sibling `GameObject`s is active based on cumulative Brain Power vs. `GameManager.RankDefinitions`, mirroring the same threshold-walk logic as `GetRankName`.

**`PlayerTapHandler`** (`Core/PlayerTapHandler.cs`) is the only player input entry point (`OnTap()`, wired to a full-screen invisible Button's `OnClick`). Each tap adds `baseTapBrainPower * tapMultiplier` Brain Power. `tapMultiplier` starts at 1 and is permanently, additively bumped +5% by `RebirthManager` on every Rebirth (`AddTapMultiplier`, added 2026-06-21 — see the rebalance section below for why) — `SetTapMultiplier` (absolute replace, used for restoring a save and for any future temporary-boost mechanic) and `AddTapMultiplier` (permanent incremental bonus) are deliberately separate methods, don't conflate them. Exposes a self-bootstrapping `Instance` (`FindAnyObjectByType`, no auto-creation — a missing instance means the tap button genuinely isn't in the scene) so `RebirthManager`/`SaveManager` can reach it without an Inspector reference. Persisted via `SaveManager`'s `tapMultiplier` field; `LoadGame()` treats a deserialized value of exactly `0` as "save predates this field" and resets it to `1`, since `JsonUtility` zero-fills missing fields rather than applying `CreateDefaultData`'s default. Calls `PlayerCharacterController.Instance.NotifyTap()` for the character's tap reaction, punch-scales an optional `tapButtonVisual` via the new `AnimationController.PlayButtonPunch` (separate from the character's own squash/stretch — no effect if unwired), and triggers `AnimationController.PlaySplatParticles` + the new `PlayFloatingRewardText` (a transient `"+X BRAIN POWER"` that rises and fades) at the tap location on an optional `particleContainer` (falls back to the first Canvas found in the scene). It no longer owns any direct tap/idle animation calls on a visual target itself — that responsibility moved entirely to `PlayerCharacterController` (see below) so there's a single owner of "what animation plays for the character in which state." Also calls `PlayerIQManager.RestoreIQFromTap()` every tap (no-op once `PlayerIQ` is back at 100). Fires `OnTapRewardEarned(double)` after every successful tap purely so other systems can react without Core reaching into UI/Systems directly — consumed by `HUDController`'s IQ-text flash (`AnimationController.PlayIQFlash`) **and** `DialogueManager`'s tap-counting for the `TapWithoutPurchase` trigger (see below). **Pointer position is read via the new Input System** (`Pointer.current.position.ReadValue()`, falling back to screen-center if no pointer device exists) — this project's `activeInputHandler` is set to Input System Package (New) only, so legacy `Input.mousePosition` throws `InvalidOperationException` at runtime. `UI/UIBlockDebugger.cs` still uses legacy `Input.GetMouseButtonDown`/`Input.mousePosition` and was *not* touched (per earlier session notes, it "appeared mid-session from elsewhere, not mine, left alone") — it will throw the same exception wherever it actually runs; flagged, not fixed, pending an explicit ask.

**`PlayerCharacterController`** (`Systems/PlayerCharacterController.cs`) is a self-bootstrapping singleton driving the on-screen Player Character's behavioral state machine: `PlayerCharacterState { Idle, Bored, Tapping, Excited }`, hand-rolled as a plain enum + coroutines (matching `AnimationController`'s established style, no Unity Animator Controller asset). `Idle`/`Bored`/`Excited` are ambient loops handed to `AnimationController` (`PlayIdleBreathing`, the new `PlayBoredFidget`, the new `PlayExcitedBounce`); `Tapping` is the existing one-shot `PlayTapAnim` squash/stretch. Transitions: `NotifyTap()` (called from `PlayerTapHandler.OnTap()`) always wins immediately and restarts a short return-to-`Idle` timer; sitting in `Idle` untouched for `boredAfterIdleSeconds` (default 20s) auto-transitions to `Bored`; `TriggerExcited()` fires from `UpgradeManager.OnBuildingPurchased`, `RebirthManager.OnRebirthCountChanged`, and `PlayerIQManager.OnIQMilestoneCrossed` (independent subscriptions, same by-convention-not-shared-implementation pattern as the IQ-milestone celebration elsewhere), then also falls back to `Idle` after a hold timer. `OnStateChanged` is a plain C# `event Action<PlayerCharacterState>`, matching the codebase's prevailing convention (not a `UnityEvent`, unlike `COGSPortraitController.OnStageChanged`/`DialogueManager.OnDialogueLine` which deviate per their own specs). Also resolves an **independent** appearance progression keyed on `RebirthCount` via `CharacterAppearanceStage` (`Systems/CharacterAppearanceStage.cs`, a `ScriptableObject` mirroring `COGSStage`'s `sprite`/`minRebirthCount` shape but deliberately a separate asset type/list — the world character and the COGS narrator portrait are distinct visual identities per spec, even though both happen to key off `RebirthCount`). No `CharacterAppearanceStage` assets exist yet (no character art) — the type exists so the upcoming Outfit/Wardrobe system has a data hook to extend.

**`WardrobeManager`** (`Systems/WardrobeManager.cs`) is a self-bootstrapping singleton owning which `OutfitData` is currently equipped on the Player Character. Unlock is **derived live**, not separately tracked: `IsUnlocked(outfit)` just checks `RebirthManager.Instance.RebirthCount >= outfit.minRebirthCountToUnlock` on demand — no persisted "owned outfits" set is needed since RebirthCount only ever increases, so once unlocked an outfit is permanently unlocked. The one piece of real, persisted state is `EquippedOutfit`, since *which* unlocked outfit is worn is a player choice (per spec — outfits are auto-unlocked like everything else, but the player picks which to wear, unlike `CharacterAppearanceStage`'s pure auto-progression). `TryEquipOutfit(outfit)` validates unlock before equipping; `hasManuallyEquipped` gates a one-time-until-overridden default (`AutoEquipLowestUnlockedIfNoneChosen`, run on `Start()` and every `OnRebirthCountChanged` until the player or a save makes a real choice) so a fresh character isn't left bare. Purely cosmetic — no gameplay effect, confirmed explicitly. `OutfitData` (`Systems/OutfitData.cs`) carries a stable `outfitId` string (not an object reference) as its save key, the same reason `UpgradeManager` keys building ownership by `buildingName`. 6 assets exist under `Assets/_Game/Outfits/` at the same RebirthCount tier breakpoints as `COGSStage` (0/1/3/6/11/20), no art yet. `WardrobeUIController`/`WardrobeSlotUI` (UI layer) mirror `ShopUIController`/`UpgradeSlotUI`'s build-one-row-per-template-and-refresh pattern exactly, with locked/unlocked-unequipped/equipped states instead of locked/affordable/too-expensive. Persisted via `SaveManager`'s `equippedOutfitId` field.

**`WorldRestorationManager`** (`Systems/WorldRestorationManager.cs`) is a self-bootstrapping singleton driving World Restoration: the dystopia-to-utopia visual transformation of the game's background, paid for by spending Points. A separate, **additive** progression layer on top of `PlayerIQ` — confirmed explicitly as an intentional revival, not a reintroduction of the old, fully-removed `WorldRestorationScore` system. `TrySpendPointsOnRestoration(amount)` calls `CurrencyManager.SpendPoints` and, on success, permanently increases `CumulativePointsSpentOnRestoration` and re-resolves the active stage. Stages (`WorldRestorationStage`, a `ScriptableObject` with `stageIndex`/`stageName`/`pointsRequired`) are resolved the same threshold-walk way as `COGSPortraitController`/rank, but the *visual* swap mirrors `DioramaManager` instead: `restorationStageObjects` (sibling `GameObject[]`, index = `stageIndex`) cross-fade via `SpriteRenderer` alpha in `Update()`, same `fadeSpeed`-per-second mechanism as `DioramaManager`'s rank backdrops. `RestorationPercent` (0–100, cumulative spent ÷ final stage's `pointsRequired`) backs `ChapterUnlockConditionType.WorldRestorationPercent` in `ChapterManager` — previously dead/always-false; now wired for real. 6 `WorldRestorationStage` assets exist under `Assets/_Game/Restoration/` (Toxic Wasteland → Utopia Achieved, thresholds 0/2,500/10,000/50,000/250,000/1,000,000 — tuned design values, rescaled 10x on 2026-06-21, see rebalance section below; superseded an earlier 0/250/1,000/5,000/25,000/100,000 pass). `HUDController` exposes a `restorationProgressText` readout and a `restoreButton` ("RESTORE" — spends all current Points at once, mirroring the existing CONVERT button's "convert everything on demand" pattern). Persisted via `SaveManager`'s `worldRestorationPointsSpent` field.

**`RebirthManager`** (`Systems/RebirthManager.cs`) is a `{ get; private set; }` singleton. `TriggerRebirth()` increments `RebirthCount` (fires `OnRebirthCountChanged`, consumed by `HUDController` and `DialogueManager`), calls `UpgradeManager.ResetBuildings()`, then `CurrencyManager.ExecuteRebirth(0.05, 0.1, 0.05)` plus `PlayerTapHandler.Instance?.AddTapMultiplier(0.05)` — flat, stacking per-rebirth bonuses to all four tap/currency levers at once: +5% Brain Power production multiplier, +10% Cash multiplier, +5% Points conversion rate, +5% tap multiplier (none scaled by how much was earned) — then calls `GameManager.Instance.RequestSave()` immediately. The tap-multiplier bonus was added 2026-06-21 specifically because manual tapping was found to lose almost all relevance to idle income within the first hour of play (see rebalance section below) — a small permanent per-rebirth bump keeps tapping from becoming a complete non-factor across many resets. **`RebirthUIController`** (`UI/RebirthUIController.cs`) owns the confirm/cancel modal *and* a separate `rebirthTriggerButton` GameObject reference — the actual "REBIRTH" button in the HUD that opens the modal. That trigger button is hidden in `Awake()` and only revealed once `WorldRestorationManager.CumulativePointsSpentOnRestoration >= pointsSpentUnlockThreshold` (default **50,000**, raised 50x from an earlier 1,000 on 2026-06-21 — see rebalance section below) — re-checked on every `OnRestorationProgressChanged`, never re-hidden since that progress is monotonic. Rebirth itself is otherwise fully functional from game start; this only gates the button's *visibility*, not the mechanic.

**`RandomEventManager`** (`Systems/RandomEventManager.cs`) periodically (90–180s, via `GameManager.OnSecondTick`) selects a `BrainRotEventData` and fires `OnRandomEventTriggered` for UI to react to; `ApplyEventEffects(eventData)` (called separately, e.g. on a popup confirm button) applies `brainPowerRewardOrPenalty` to Brain Power and `playerIQImpact` to `PlayerIQ`, then fires `OnEventResolved` (only on accept, not on decline — consumed by `DialogueManager`'s `EventOutcome` trigger). 8 `BrainRotEventData` assets currently exist under `Assets/_Game/Events/`.

**`SaveManager`** (`Systems/SaveManager.cs`, `[DefaultExecutionOrder(-200)]` so it loads before `GameManager`'s `-100`) persists a `PlayerData` struct (`currentBrains`, `cumulativeBrains`, `rebirthMultiplier`, `rebirthCount`, `playerIQ`, `buildingLevels`, plus `currentCash`/`cashMultiplier`/`currentPoints`/`pointsConversionRate`/`autoConvertCash` for the Cash/Points tiers — `autoConvertCash` wasn't in the original Cash/Points spec but was added anyway since an unpersisted player-facing toggle would silently reset every session) as JSON to `Application.persistentDataPath/braindrain_save.json`. `LoadGame()` runs in `Awake()`; `ApplyLoadedDataToSystems()` (in `Start()`) is what actually pushes the loaded snapshot into the four managers' `LoadState`/`LoadBuildingLevels` methods — `LoadGame()` alone only populates the in-memory struct. `SaveGame()` is hooked to `GameManager.OnSaveRequested`, which three independent triggers call: `OnApplicationPause(true)`, `OnApplicationFocus(false)`, `OnApplicationQuit()` (all on `GameManager` itself), plus a 60-second counter folded into `GameManager.ProcessSecondTick()` rather than a second `InvokeRepeating`. `DeleteSave()` exists for future debug/reset use. Note `cashPerSecond`/`idleBpps` are deliberately *not* in `PlayerData` — both are re-derived from `buildingLevels` on load (see `UpgradeManager.LoadBuildingLevels`), not saved directly, to avoid a second source of truth that could drift from actual building ownership.

**`AnimationController`** (`Systems/AnimationController.cs`) is mostly a coroutine-based singleton exposing static methods other scripts call directly: `PlayTapAnim`, `PlayIdleBreathing`/`StopIdleBreathing`, `PlayBoredFidget`/`StopBoredFidget` (+/-4° Z-tilt wobble) and `PlayExcitedBounce`/`StopExcitedBounce` (1.0↔1.12 scale bounce) — both added for `PlayerCharacterController`'s `Bored`/`Excited` states and following the exact same Transform-keyed coroutine-dictionary pattern as `PlayIdleBreathing` — `PlaySplatParticles` (procedurally-generated placeholder pink circle sprite, no art asset needed), `PlayTouchRipple` (neon ring + glow), `PlayAffordablePulse`/`StopAffordablePulse` (wired into `UpgradeSlotUI`, only on actual affordable-state transitions), `PlayPopupSpawn` (wired into `RebirthUIController`/`RandomEventUIController`), and `PlayHighIQCelebration` (triggered by `HUDController` every 1000 `PlayerIQ`). **DOTween is now in the project** (`Assets/Plugins/Demigiant/DOTween`, imported 2026-06-21) — `PlayButtonPunch` (`DOPunchScale`), `PlayFloatingRewardText` (`DOAnchorPos` + `DOFade`), and `PlaySlide` (`DOAnchorPos`) were migrated from hand-rolled coroutines to it for smoother curves; everything else intentionally stayed on coroutines (simple one-shots/loops don't need a tweening engine). The three migrated methods use `Dictionary<T, Tween>` + a `KillExisting` helper as the DOTween-native equivalent of `StopAndReplace`'s kill-and-replace semantics.

**`DialogueManager`** (`Systems/DialogueManager.cs`) listens for 8 triggers — the original 5 (`OnFirstBrainPowerEarned`, `OnBuildingPurchased`, `OnRebirthCountChanged`, `OnEventResolved`, `OnIQMilestoneCrossed`) plus 3 added 2026-06-21 (`CurrencyManager.OnCashConverted` → `CashConverted`; `PlayerIQManager.OnOfflineDecayApplied` → `OfflineDecayReturn`; and `PlayerTapHandler.OnTapRewardEarned`, counted internally as `tapsSinceLastPurchase` and fired as `TapWithoutPurchase` every 10 taps, reset to 0 by any `OnBuildingPurchased` — so it only fires during stretches with zero purchases, never as a flat "every 10th tap" counter) — filters its `NarratorLine` pool by `triggerType` + current `RebirthManager.RebirthCount` falling within `[minRebirthCount, maxRebirthCount]` (+ `buildingName` for `BuildingPurchase`), picks one at random (never repeating the immediately-previous line), and fires a `UnityEvent<string>` called `OnDialogueLine` (a concrete `DialogueLineUnityEvent` subclass, per spec — note this is the one place in the codebase using `UnityEvent` instead of the prevailing C# `event Action<T>` convention). Queues up to 2 lines while one is displaying; drops anything beyond that. The 30 seeded `NarratorLine` assets under `Assets/_Game/Dialogue/` were originally gated on `PlayerIQ` (which only ever increases and can't represent a degrading tone — see git history); regated onto `RebirthCount` instead: 0 (smug corporate) / 1–2 (buzzword salad) / 3–5 (broken grammar) / 6–10 (mostly enthusiasm) / 11+ (pre-collapse). The `IQMilestone` *trigger* itself is unchanged (still fires from `PlayerIQManager.OnIQMilestoneCrossed` on every 1000-point crossing) — only the line-selection filter changed, so an `IQMilestone` event now picks a line based on the player's current rebirth tier rather than IQ. **`DialogueDisplayUI`'s font size/letter-spacing degradation is still keyed off `PlayerIQ` directly** (clean at 100, chaotic by 20) and was *not* part of this regating — since `PlayerIQ` keeps climbing indefinitely, the visual font effect will likely stay "clean" even when a pre-collapse-tier (`RebirthCount` 11+) line is being displayed. That's a now-loose inconsistency between line content and line presentation, not yet reconciled.

11 new `NarratorLine` assets seed the 3 new triggers (`Assets/_Game/Dialogue/FirstTap_Awake`, `BuildingPurchase_*` ×7 — one per building, `buildingName`-matched, gated `[0, int.MaxValue]` so they apply at every Rebirth tier rather than being tone-tiered like the original 30 — `TapWithoutPurchase_BuySomething`, `CashConverted_VeryResponsible`, `OfflineDecayReturn_WelcomeBack`). **`FirstTap_Awake` is a second candidate at the same `[0,0]` tier as the existing `Tier80to100_FirstTap`** — since `FirstTap` only ever fires once per save (ever), there's now a coin-flip between the two possible lines rather than a guaranteed match; the old line wasn't removed since pool variety is the system's whole design intent, but if a guaranteed match matters more than variety here, that's a one-line deletion away.

**`COGSWorldPortraitUI`** (`UI/COGSWorldPortraitUI.cs`) is a small, separate listener on `COGSPortraitController.OnStageChanged` (same subscription pattern as `DialogueDisplayUI`'s own avatar) that keeps a **persistent** world-area `Image` in sync with COGS's current stage — distinct from `DialogueDisplayUI.avatarImage`, which only exists inside the transient dialogue panel and slides in/out with each line. This is meant to be COGS's actual on-screen presence (always visible), while the dialogue panel is the separate, transient speech-bubble overlay that pops up when she "talks." Both coexist; this doesn't replace the dialogue panel's own avatar slot.

**UI layer** (`BrainDrain.UI` namespace) is purely reactive: `HUDController` and `ShopUIController` subscribe to Core events (`OnBrainPowerChanged`, `OnCumulativeBrainPowerChanged`, `OnPlayerIQChanged`, `OnBuildingsChanged`, `OnRebirthCountChanged`) in their own `Start`/`OnGameInitialized` handlers and unsubscribe in `OnDestroy` — always follow this subscribe-in-init/unsubscribe-in-destroy symmetry when adding new UI. `HUDController.OnCashChanged`/`OnPointsChanged` subscriptions use `AddListener`/`RemoveListener` instead, since those two are `UnityEvent`s (see `CurrencyManager` above) — don't mix up the two patterns when adding new Cash/Points UI. `HUDController` also pulls `CurrencyManager.IdleBPPS` directly on every `GameManager.OnSecondTick` for its BPPS readout, rather than via a dedicated push event (idle BPPS itself only changes at purchase/reset time, not every tick) — Cash's CPS readout instead piggybacks on `OnCashChanged`, which already fires every tick once `CashPerSecond > 0`. `HUDController`'s "CONVERT" button calls `ConvertCashToPoints(CurrentCash)` (convert everything on demand); there's currently no UI for toggling `AutoConvertCash`. `ShopUIController` builds one `UpgradeSlotUI` instance per `BuildingData` template at runtime (no pre-authored rows), **sorted by `unlockCumulativeBrainPower` ascending** (cheapest/earliest-unlocking first) regardless of `buildingTemplates`' Inspector order — `transform.SetSiblingIndex(i)` is forced on every slot, reused or freshly instantiated, so rebuilds can't leave stale ordering; `UpgradeSlotUI` renders three visual states (locked / affordable / too-expensive) using fixed neon hex colors and calls back into `UpgradeManager.TryBuyBuilding` on click. **`ShopUIController` also now owns the shop's popup panel visibility** (`shopPanel`/`shopButton`/`closeButton`) — hidden in `Awake()` by default, opened/closed via `OpenShop()`/`CloseShop()`. Unlike `RebirthUIController`'s modal (a shake-in `PlayPopupSpawn`), the shop panel **slides up from offscreen-below** into its authored resting position on open and slides back down before deactivating on close, via the new generic `AnimationController.PlaySlide(rect, from, to, duration, onComplete)` primitive — `ShopUIController.Awake()` captures `shopPanelRect.anchoredPosition` *before* hiding the panel, since that's the only reliable way to know "resting position" vs. "offscreen," and `CloseShop()` passes `onComplete` to `SetActive(false)` only once the slide-down actually finishes (the coroutine runs on `AnimationController`'s own always-active GameObject, so hiding `shopPanel` mid-slide doesn't interrupt it). This is a deliberate layout change: buildings used to occupy the main game view permanently; now they're a popup the player opens on demand, freeing that screen space for the Player Character / World Restoration backdrop / COGS dialogue panel to coexist there instead. `content` (the scrollable rows container) is distinct from `shopPanel` (the whole popup, shown/hidden as a unit) — building/refreshing rows works fine while `shopPanel` is inactive, since nothing in `UpgradeSlotUI` depends on `Start()`/being active. Locked buildings already showed `"??? CLASSIFIED ???"` plus the exact unlock requirement (`costText` → `"{X} BRAIN POWER REQUIRED"`) before this change — confirmed working as designed, not touched.

**Important gotcha, hit for real on 2026-06-21:** `ShopUIController` lives *on* the `ShopPanel` GameObject itself. The Shop popup's "hidden by default" state must come from `ShopUIController.Awake()` calling `SetActive(false)` on itself *after* registering `shopButton`/`closeButton`'s `onClick` listeners — **never** from saving `ShopPanel` as inactive in the scene file directly. Unity only calls `Awake()` on a GameObject that's active at scene load (or the first time it's later activated) — if the saved scene has it inactive from the start, `Awake()` never runs at all, so the `onClick.AddListener` calls never happen either, and both buttons silently do nothing forever. `Editor/ShopPanelLayoutFix.cs` used to call `shopPanel.SetActive(false)` itself after wiring, which baked exactly this bug into the saved scene — removed; it now leaves the panel active and trusts `ShopUIController.Awake()`'s own (correct) deactivation. `Editor/AutoSceneFixes.cs` also auto-reactivates `ShopPanel` on every script reload as a safety net against this regressing again.

**Known gap, now fixed by `Editor/ShopPanelLayoutFix.cs` (run via `BrainDrain > Fix Shop Panel Layout`):** the code above shipped correctly, but the scene-side `shopPanel`/`shopButton`/`closeButton` fields on the actual `ShopPanel` GameObject's `ShopUIController` component were left unwired (`fileID: 0`) — so the panel just sat permanently active at its originally-authored anchors `(0,0.45)-(1,0.875)`, eating the main view, with nothing controlling it. The Editor tool resizes it to the bottom 60% (`(0,0)-(1,0.6)`), builds a cyan "SHOP" button styled to match `ConvertButton` (same rounded sprite/font/`LayoutElement` sizing) inserted into the `EconomyBar` `HorizontalLayoutGroup` row immediately left of Convert, builds a close/X button anchored top-right of the panel, wires all three fields, hides the panel, and saves the scene. Idempotent (finds and updates existing `ShopButton`/`CloseButton` GameObjects by name on re-run rather than duplicating). `NumberFormatter.Format(double)` is the shared idle-game number formatter (e.g. `1.25M`) — use it for any new currency display rather than formatting doubles directly.

## Economy rebalance (2026-06-21)

A numeric pacing simulation (`balance_sim.js`, project root, outside `Assets/` — disposable, not part of the game) modeled the real tuned values against a greedy "always buy cheapest unlocked+affordable building" AI to sanity-check pacing without needing a live Editor session. It surfaced three problems, all addressed here:

1. **The whole pre-Rebirth building ladder cleared in under an hour**, and the Rebirth button itself (gated on Points spent on World Restoration) unlocked around 30 minutes in. Fix: `pointsSpentUnlockThreshold` on `RebirthUIController` raised **1,000 → 50,000**, and all 7 buildings' `costMultiplier` raised **1.15 → 1.21** (`Assets/_Game/Buildings/*.asset`). Re-simulated at 24 hours of *continuous, optimal, 3-taps/sec* play: Rebirth now unlocks around the 5-hour mark under that unrealistically relentless assumption — comfortably beyond a real player's first day, since nobody taps nonstop for 5 hours and idle progress doesn't accrue while the app is fully closed (no offline-earnings batch calculation exists for currency, only for `PlayerIQ` decay).
2. **Manual tapping lost almost all relevance to idle income within the first 15 minutes** (tripling tap rate changed 4-hour outcomes by only ~3-5%). Fix: `PlayerTapHandler.tapMultiplier` (previously a static, never-modified 1.0) now gets a permanent +5% bump from `RebirthManager` on every Rebirth via the new `AddTapMultiplier` method, mirroring the existing Brain Power/Cash/Points rebirth bonuses. Persisted via the new `SaveManager`/`PlayerData.tapMultiplier` field (with a load-time migration guard treating a deserialized `0` — i.e. a save predating this field — as "use the 1.0 default", since `JsonUtility` zero-fills missing fields rather than applying `CreateDefaultData`'s default).
3. **World Restoration's six stage thresholds didn't represent a believable long-term arc** once the above two were fixed. All 6 `WorldRestorationStage.pointsRequired` values rescaled 10x: 0/2,500/10,000/50,000/250,000/1,000,000 (was 0/250/1,000/5,000/25,000/100,000). The final stage, Utopia Achieved, isn't reached even after 24 simulated hours of continuous optimal play — a deliberate "and beyond" long-term goal, not a same-session finish line.

A fourth, related change has no direct numeric target from the simulation but addresses a real mechanic gap it exposed: **idle BPPS/CPS production is now scaled by current `PlayerIQ`** (`CurrencyManager.GetIQProductionMultiplier()`, `PlayerIQ / 100` clamped to a max of 1, applied in `HandleSecondTick` before crediting Brain Power/Cash each tick). Previously a decayed `PlayerIQ` (from the offline-decay-on-load hook, floor 60) had zero gameplay effect — purely cosmetic/narrative. Now, returning from being offline with decayed IQ means idle income comes back at a reduced rate (as low as 60% at the decay floor) until tapping restores `PlayerIQ` back to 100 — so being offline no longer gives a full, undiscounted 100% idle gain the instant the app reopens. Tap income itself is deliberately untouched by this multiplier, since tapping is the only way to recover IQ in the first place.

## Illumisnotti narrative/economy rewrite (2026-06-21)

A large additive pass layering Shop 2 (Cash), Shop 3 (Points), a God Tier Store (real money), a timed-effect framework for chaos events, and "The Snotting" rename on top of the existing Brain Power/Cash/Points/Rebirth economy. Built per an explicit, fully-specified design request. **Nothing existing was removed or had its signature changed** — every hook below defaults to a neutral no-op (multipliers start at 1.0/0.0, gates default open) so a save with zero engagement with any of this behaves identically to before the pass.

**Currency loop, unchanged, just confirmed:** Brain Power → Shop 1 (existing 7 buildings, now with Illumisnotti-flavored `description` text on 3 of them: Literal Library, Doomscroll Engine, Underground Economy) → generates passive Cash → Shop 2 *or* converted to Points → Shop 3 → world domination. Real Money is a separate, parallel rail feeding only the God Tier Store.

**Shop 2 — Cash Shop** (`Systems/CashShopItemData.cs`, `Systems/CompanionTierData.cs`, `Systems/CashShopManager.cs`, `Systems/CompanionManager.cs`, `UI/CashShopUIController.cs`, `UI/CashShopSlotUI.cs`): two independent pieces of state. `CashShopManager` owns 5 generic one-time items (Golden Cardboard Crown, Shivering Designer Micro-Dog, Overpriced Branded Sneakers, Private VIP Velvet Rope, Solid Gold Gaming Throne), each gated only on `RebirthCount` and applying one of three effect types (`BrainPowerTapPercent` → `PlayerTapHandler.AddTapMultiplier`, `CashPerSecondPercent` → `CurrencyManager.AddShopCashMultiplierBonus`, `AllMultipliersPercent` → both `AddShopAllMultiplierBonus` and `AddTapMultiplier`). `CompanionManager` separately owns the Hot Chick companion — a single evolving sprite/quote across 6 *strictly sequential* tiers (`CompanionTierData`, tier N requires already owning tier N-1), each granting a permanent Cash-per-second percent via the same `AddShopCashMultiplierBonus`. Tier 1 is gated on 24 real-world hours since first launch (`SaveManager.FirstLaunchUnixSeconds`, a new persisted field set once on the very first ever save and never touched again); tiers 2-5 gate on `RebirthCount`; tier 6 gates on `RebirthCount` *and* a World Restoration stage index (`CompanionGateType.RebirthCountAndWorldStage`).

**Shop 3 — Points Power Shop** (`Systems/PointsShopItemData.cs`, `Systems/PointsShopManager.cs`, `UI/PointsShopUIController.cs`, `UI/PointsShopSlotUI.cs`): 6 one-time items gated on `RebirthCount` and optionally a World Restoration stage index (`gateWorldStageIndex = -1` means no stage gate). Four effect types: `PointsConversionRatePercent` (→ new public `CurrencyManager.AddPointsConversionRateBonus`, the same mechanism `ExecuteRebirth` already used internally, now exposed), `CashToPointsConversionPercent` (→ `AddShopCashToPointsBonus`, a *distinct* multiplier from the conversion rate itself — deliberately kept separate so "The Snotty Guard" and the rate-bonus items stack independently rather than colliding on the same number), `AllPointGainsPercent` (→ `AddShopAllPointGainsBonus`, applied in `ConvertCashToPoints` alongside the Cash-to-Points one). The 6th item, **The Grand Snotting**, is the capstone: `GrandSnottingCapstone` is the one effect type that's a literal `10x` *multiplicative* jump (`CurrencyManager.MultiplyShopAllMultiplier(10d)` plus a matching `PlayerTapHandler.SetTapMultiplier` jump), not an additive percent like the other 5, and sets `PointsShopManager.SecretEndingUnlocked` permanently — **no actual ending sequence/UI exists yet**; the flag exists for a future one to check.

**God Tier Store — real money** (`Systems/GodTierStoreItemData.cs`, `Systems/GodTierStoreManager.cs`, `UI/GodTierStoreUIController.cs`, `UI/GodTierStoreSlotUI.cs`): **NO real payment processing exists in this project** — no Unity IAP package, no App Store/Play Store product IDs configured. `GodTierStoreManager.StubPurchase` is a clearly-marked placeholder that grants the item immediately for free; wire a real IAP plugin's purchase-success callback to call it instead of a bare UI button before ever shipping this, or anyone can "buy" these items at no cost. 4 of the 5 items are pure cosmetic flags with no other code consuming them yet (`CogsVoicepackDisdainOwned`, `Y2KGlitchSlumThemeOwned`, `IllumisnottiMembershipCardOwned`, `HolographicTrashCanFlexOwned` — flags exist, no actual voicepack/theme swap is implemented). The one item with a real mechanical effect is the **24-Hour Corporate Cloak**: `PlayerIQManager.ExtendOfflineDecayWindow(hours)` (new) permanently adds to `bonusOfflineDecayMaxHours`, extending the existing offline-decay-to-floor-60 window past its base 8 hours — "convenience not power" per its own description, since it doesn't change the floor or `PlayerIQ` itself, only how long you have before reaching the existing floor. **Load-order dependency**: `SaveManager.ApplyLoadedDataToSystems` calls `GodTierStoreManager.LoadState` *before* `PlayerIQManager.LoadStateWithOfflineDecay`, specifically so an owned Cloak's bonus hours are in place before that load's decay calculation runs — getting this order backwards would under-credit the bonus for exactly one load.

**Timed-effect framework for Illumisnotti random events** (`Systems/BrainRotEventData.cs` extended, `Core/CurrencyManager.cs`, `Core/PlayerTapHandler.cs`, `Core/UpgradeManager.cs`, `Systems/RandomEventManager.cs`): the original `BrainRotEventData`/`RandomEventManager.ApplyEventEffects` only ever applied instant Brain Power/IQ deltas. Added a new `EventEffectType` enum (`InstantOnly` — the original 8 events' implicit, unchanged behavior — `LockRandomBuilding`, `FreezeTap`, `BrainPowerProductionPercent`, `AllMultipliersPercent`, `RandomBuffOrDebuff`) plus `effectDurationSeconds`/`effectMagnitudePercent`/`cashRewardOrPenalty` fields (the last because no instant-Cash-reward pathway existed before — `Illumisnotti Leak` needed one). Mechanisms, all reversible/lazily-expiring rather than coroutine-driven:
- `CurrencyManager.SuppressIdleBPPS`/`RestoreIdleBPPS`/`SuppressCashPerSecond`/`RestoreCashPerSecond` — reversible deltas to the aggregate idle totals, distinct from `AddIdleBPPS`'s permanent purchase-time grant.
- `UpgradeManager.LockRandomBuildingFor(seconds)` — picks one currently-*owned* building at random, computes its exact current contribution (`level * baseBPPS`/`baseCPS`), suppresses exactly that amount, and restores it on a later `GameManager.OnSecondTick` via a small internal `(bpps, cps, restoreAtTime)` list. `UpgradeManager` did not previously subscribe to the tick at all; it now does, lazily, only once the first lock is ever applied.
- `CurrencyManager.ApplyTemporaryBrainPowerModifier`/`ApplyTemporaryAllMultiplierModifier` and `PlayerTapHandler.ApplyTemporaryTapModifier`/`FreezeTapsFor` — store a `(percent, expiryTime)` pair, lazily expiring (checked and zeroed on next read inside `AddBrainPower`/`AddCash`/`OnTap`) rather than via a dedicated cleanup tick, since those methods already fire constantly. `BrainPowerProductionPercent` only flows through `AddBrainPower` (hitting both tap and idle BPPS, since they share that method); `AllMultipliersPercent` additionally hits `AddCash` and gets a matching tap-side call, so it's strictly broader than the BP-only variant — this distinction is intentional, matching "BP drops" (Propaganda Broadcast) vs. "all multipliers" (Lord Snott Tantrum) in the source spec.
- `RandomBuffOrDebuff` (The Snotty Newsletter) rerolls its sign at apply time in `RandomEventManager` — author `effectMagnitudePercent` as a positive magnitude only.

8 new `BrainRotEventData` assets exist under `Assets/_Game/Events/` (`IllumisnottiMinistryInspection`, `IllumisnottiSnottTaxAudit`, `IllumisnottiPropagandaBroadcast`, `IllumisnottiBrawndoSpill`, `IllumisnottiCeaseAndDesist`, `IllumisnottiLordSnottTantrum`, `IllumisnottiLeak`, `IllumisnottiSnottyNewsletter`) **added to**, not replacing, the original 8 — `RandomEventManager.potentialEvents` needs all 16 dragged in via Inspector (see "Pending scene wiring" below).

**The Snotting** (`Systems/RebirthManager.cs`, `UI/RebirthUIController.cs`, `UI/HUDController.cs`): deliberately a **player-facing text rename only** — `RebirthManager`, `RebirthCount`, `TriggerRebirth`, `OnRebirthCountChanged` etc. keep their existing C# names throughout the codebase (an invasive rename across every system already referencing them was judged not worth the risk for a cosmetic change). New `RebirthManager.GetIllumisnottiTitle(int rebirthCount)` (static) maps tier → title: 0 → none, 1 → "Junior Associate Snott", 2 → "Regional Snott Manager", 3 → "Vice President of Snottery", 4 → "Lord Snott (Provisional)", 5 → "Grand Illumisnotti", 6+ → "Supreme Snott Eternal". `HUDController` gained `illumisnottiTitleText` (new optional serialized field, **not yet wired to a scene GameObject** — see below), updated alongside the existing `rebirthCountText` (now reads "SNOTTINGS: {count}" instead of "REBIRTHS: {count}") on every `RebirthManager.OnRebirthCountChanged`. The HUD's actual "REBIRTH" button label was hand-edited directly in `SampleScene.unity` to read "THE SNOTTING" (a single `TextMeshProUGUI.m_text` value — the kind of simple field tuning this project's hand-editing convention already covers, not new GameObject construction).

**Pacing numbers were deliberately reused from the 2026-06-21 economy rebalance, not re-derived** — e.g. Hot Chick tier 6 / Grand Snotting both gate partly on `RebirthCount: 7`, well past where that rebalance pass calculated Rebirth itself becoming reachable (~5 hours of continuous optimal play, translating to multiple real days for an actual player) — so these endgame items land naturally late without needing their own separate pacing pass.

**Asset locations, all confirmed written and GUID-clean against the rest of the project:** 5 `CashShopItemData` under `Assets/_Game/CashShop/`, 6 `CompanionTierData` under `Assets/_Game/CashShop/Companion/`, 6 `PointsShopItemData` under `Assets/_Game/PointsShop/`, 5 `GodTierStoreItemData` under `Assets/_Game/GodTierStore/`. 40 new `NarratorLine` assets under `Assets/_Game/Dialogue/Illumisnotti/` (additive to, not replacing, the original 30) seed `FirstTap`/`BuildingPurchase`/`Rebirth`/`EventOutcome`/`IQMilestone`/`CashConverted`/`TapWithoutPurchase` with Illumisnotti lore across the same 5 `RebirthCount` tone bands as the original pool (`[0,0]`/`[1,2]`/`[3,5]`/`[6,10]`/`[11,maxint]`), 8 per band — none of them set `buildingName`, so unlike the 7 building-specific `BuildingPurchase_*` lines from the earlier pass, these apply to a purchase of any building. `DialogueManager.narratorLines` needs all 40 (plus the 30 original) dragged in via Inspector, same caveat as `RandomEventManager.potentialEvents` below.

## Editor-only progression testing system

Added 2026-06-21, entirely `#if UNITY_EDITOR` — every piece below compiles out completely in any build, even a Development Build, since `UNITY_EDITOR` is only ever defined inside the Editor itself.

- **`Systems/DebugCheats.cs`** — the actual cheat logic, exactly once, called by both the in-game panel and the Editor menu. Lives in `Systems/` (not `Editor/`) specifically because `DebugCheatPanel` (a runtime `MonoBehaviour`) needs to reference it, and scripts inside an `Editor/` folder compile into an Editor-only assembly that runtime scripts can't see. `MaxAllBuildings()` deliberately does **not** call `UpgradeManager.LoadBuildingLevels` directly — that method replays `AddIdleBPPS`/`AddCashPerSecond` additively with no reset, so calling it on top of already-owned buildings would double-count BPPS/CPS. Instead it grants exactly each building's current cost via `AddBrainPower` (which also raises `CumulativeBrainPower`, naturally unlocking higher tiers) immediately before each `TryBuyBuilding` call, routing through the same real purchase pathway normal play uses. `ClearSave()` only touches `SaveManager.Instance` while `Application.isPlaying` — outside Play Mode it deletes the file directly via the now-public `SaveManager.SaveFilePath`, since touching a self-bootstrapping `Instance` in Edit mode would leave a permanent stray GameObject in the open scene.
- **`UI/DebugCheatPanel.cs`** — self-bootstraps via `[RuntimeInitializeOnLoadMethod(AfterSceneLoad)]` (no Inspector wiring step to leave unset — the exact failure mode that broke the Shop panel twice this session). Finds `HUDController.PlayerIQText` itself, adds a `Button` to that same GameObject if one isn't already there, and counts clicks within a 0.6s rolling window to detect a triple-tap, which toggles the panel. The panel itself (background, hot-pink border, button list) is built entirely procedurally at `Start()`, parented under the IQ text's own `Canvas`. Open/closed state persists via `EditorPrefs` (`BrainDrain_DebugPanelOpen`), independent of the save file itself.
- **`Editor/TestingMenuShortcuts.cs`** — the `BrainDrain/Testing` menu, calling the same `DebugCheats` methods. Everything except `Reset Save`/`Toggle Persistent Save` is wrapped in a `RequirePlayMode` guard, since those cheats reach into live singletons that only meaningfully exist once Play Mode has started.
- **`SaveManager`** — `KeepSaveEditorPrefsKey` (`"BrainDrain_KeepSave"`, default `true`) checked in `Awake()`: if false, deletes the save before `LoadGame()` runs, forcing a fresh start every time Play Mode begins. Also registers an `[InitializeOnLoadMethod]` hook on `EditorApplication.playModeStateChanged` that force-saves on `ExitingPlayMode` — necessary because stopping Play Mode in the Editor does **not** fire `OnApplicationQuit`/`Pause`/`Focus` the way a real device would, so without this, progress since the last 60s autosave could silently be lost on every Stop, undermining the entire point of the keep-save toggle.
- **`CurrencyManager.AddPoints(double)`** — new, general-purpose (not cheat-gated itself). Points previously had no direct "add" method, only `ConvertCashToPoints`, which made granting an exact Points amount for testing require reverse-calculating against the current rebirth-tier-dependent conversion rate.
- **`WorldRestorationManager.Stages`** — new public read-only accessor (also not cheat-gated). Lets `DebugCheatPanel`'s stage-jump buttons read real configured thresholds instead of hardcoding values that could drift out of sync if the stages are retuned later.

## Editing ScriptableObject data

Building balance values (`baseCost`, `costMultiplier`, `baseBrainPowerPerSecond`, `baseCashPerSecond`, `unlockCumulativeBrainPower`) live in the `.asset` YAML files under `Assets/_Game/Buildings/` and can be tuned by editing those fields directly. `RankDefinition[]` thresholds live on the `GameManager` component instance in `Assets/Scenes/SampleScene.unity` (not in a separate asset), so changing rank thresholds means editing the scene file's serialized array or doing it in the Editor Inspector.
